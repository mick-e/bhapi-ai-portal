------------------------------------------------------------
Categorization schema (K / F / R)
------------------------------------------------------------

Every `except Exception` in the list below must be categorized as one of:

  K  KEEP (fire-and-forget)
       The try/except is correct as-is in intent: best-effort code where a
       single failure should not cascade. Fix: replace `pass` with
         `logger.debug("<op>_degraded", exc_info=True, ...)`.
       Canonical examples:
         * SSE fan-out (src/alerts/sse.py) — one dead subscriber queue must
           not kill the broadcast.
         * Push/email side effects inside alert processing.
         * Best-effort audit writes.
         * Dashboard aggregation sections (src/portal/service.py) — each
           section wrapped so one section's failure shows a "degraded"
           banner instead of a 500. The router's outer catch-all already
           returns degraded_sections=["all"] on unexpected errors; preserve
           both layers.

  F  FIX (business logic)
       The exception indicates a real failure that should be surfaced to
       the caller or retried. Fix: either (a) narrow to a specific exception
       class (httpx.HTTPError, sqlalchemy.exc.*, etc.) and let others
       propagate, or (b) keep the broad catch but log at WARNING/ERROR and
       either re-raise or return an explicit failure sentinel.
       Canonical examples:
         * Billing writes, integration API calls where silent failure =
           lost state.
         * Token refresh paths where swallowing means the user stays
           logged out without explanation.

  R  REMOVE (unnecessary try/except)
       The wrapped code cannot actually raise Exception, or the except
       block does nothing useful (e.g. `except Exception: raise`,
       `except Exception: pass` with no side effects and the caller cannot
       observe the swallowed failure). Fix: delete the try/except; let
       Python's default behavior apply.

Replacement rules (matches plan Task 13 / R-10):

  Before                                After (K)
  ------                                ----------
  try:                                  try:
      do_side_effect()                      do_side_effect()
  except Exception:                     except Exception:
      pass                                  logger.debug(
                                                "side_effect_degraded",
                                                exc_info=True,
                                            )

  Before                                After (F)
  ------                                ----------
  try:                                  try:
      charge_customer()                     charge_customer()
  except Exception:                     except Exception:
      pass                                  logger.warning(
                                                "charge_failed",
                                                exc_info=True,
                                            )
                                            raise  # or return sentinel

All logs must use structlog's key=value args — NEVER f-strings.
Correlation IDs flow through middleware automatically.
