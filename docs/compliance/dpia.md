## 1. Processing Overview

### 1.1 What data is processed?

| Data Category | Examples | Lawful Basis |
|---|---|---|
| Account data | Email, display name, account type, hashed password | Contract (Art. 6(1)(b)) |
| Group membership | Parent-child relationships, roles | Legitimate interest (Art. 6(1)(f)) |
| AI interaction metadata | Platform (ChatGPT, Gemini, etc.), timestamp, session duration | Legitimate interest + consent for minors |
| Risk events | Flagged content categories, severity levels, PII indicators | Legitimate interest (child safety) |
| Content excerpts | Partial prompts/responses flagged by risk engine (redacted) | Consent (Art. 6(1)(a)) + Art. 9 safeguards |
| Spend data | LLM API usage costs, provider, token counts | Contract |
| Consent records | Consent type, given_at, withdrawn_at, evidence | Legal obligation (Art. 6(1)(c)) |
| Audit logs | Actor, action, resource, timestamp, IP address | Legal obligation |

### 1.2 Why is data processed?

Bhapi monitors children's use of AI tools (ChatGPT, Gemini, Copilot, Claude, Grok) to:

- Detect safety risks (self-harm, violence, bullying, academic dishonesty)
- Alert guardians/educators to concerning interactions in real time
- Enforce age-appropriate content boundaries
- Track and control AI spending
- Maintain compliance audit trails

### 1.3 Who is involved?

- **Data Controller:** Bhapi (bhapi.ai)
- **Data Subjects:** Children/minors using AI tools; guardians managing accounts
- **Data Processor:** Bhapi (self-hosted infrastructure)
- **Sub-processors:** Stripe (billing), SendGrid (email), Cloud infrastructure (Render/GCP)
- **DPO Contact:** dpo@bhapi.ai

### 1.4 How is data processed?

Data flows through a pipeline: Capture (browser extension/DNS/API) → Risk Engine (PII detection, safety classification) → Alerts → Storage → Reporting → Scheduled Deletion.

## 2. Necessity and Proportionality

### 2.1 Why is this processing necessary?

Children's unsupervised AI tool usage poses documented risks:

- Exposure to harmful content generated by AI
- Inadvertent disclosure of personal information to AI providers
- Academic dishonesty facilitated by AI
- Uncontrolled spending on AI API credits
- Psychological risks from AI-generated self-harm or bullying content

Guardian oversight is a fundamental right and obligation. Bhapi enables this oversight in the digital AI space, analogous to parental controls for web browsing.

### 2.2 Why are less invasive alternatives insufficient?

| Alternative | Why Insufficient |
|---|---|
| No monitoring | Leaves children unprotected from documented AI risks |
| Manual review by parents | Impractical at scale; parents cannot monitor every interaction |
| Platform-native controls | Inconsistent across providers; many lack child-specific safeguards |
| Full interaction logging | Disproportionate — Bhapi only flags risk events, not all interactions |
| Keyword blocking only | Insufficient — context matters for risk assessment |

### 2.3 Proportionality measures

- **Data minimisation:** Raw prompts/responses are NOT stored by default. Only risk-flagged content excerpts are retained, and these are automatically redacted for PII.
- **Purpose limitation:** Data is used solely for child safety monitoring and spend management. No advertising, profiling for commercial purposes, or data sales.
- **Storage limitation:** Risk event data auto-deletes after 12 months. Account data deletes immediately upon account deletion.
- **Accuracy:** Guardians can review and correct all flagged events.

## 3. Data Flow Mapping

```
[Browser Extension] ──HMAC-signed──> [Capture Gateway]
         │                                   │
         │                            [PII Detection]
         │                                   │
         │                         [Safety Classifier]
         │                          (keyword / Vertex AI)
         │                                   │
         │                           [Rules Engine]
         │                          (age-band scaling)
         │                                   │
         │                    ┌───────────────┼───────────────┐
         │                    │               │               │
         │              [Risk Events]   [Alerts DB]    [Email Alerts]
         │              (encrypted)     (encrypted)    (SendGrid TLS)
         │                    │               │
         │              [12-month TTL]  [Dashboard View]
         │                                   │
         │                           [Guardian Review]
         │                                   │
         │                    ┌───────────────┼───────────────┐
         │                    │               │               │
         │              [Acknowledge]   [Export (GDPR)]  [Delete (GDPR)]
         │
    [Spend Records] ──encrypted creds──> [LLM Provider APIs]
         │
    [Budget Alerts]
```

### 3.1 Data at rest

- PostgreSQL database with encryption at rest (cloud provider managed)
- LLM API credentials encrypted with Fernet (dev/test) or Cloud KMS (production)
- No raw AI interaction content stored outside of risk-flagged excerpts

### 3.2 Data in transit

- All external communication over TLS 1.2+ (HTTPS enforced via HSTS)
- Browser extension events signed with HMAC-SHA256
- Internal service communication within private network (Render)

### 3.3 Data retention

| Data Type | Retention Period | Deletion Method |
|---|---|---|
| Risk events & content excerpts | 12 months | Automated job |
| Audit log entries | 24 months | Automated job |
| Account data | Until account deletion | Soft-delete cascade |
| Consent records | Until withdrawal + 6 months | Preserved for legal compliance |
| Spend records | 12 months | Automated job |
| Session tokens | 24 hours | Auto-expiry |

## 5. Mitigation Measures

### R1: Unauthorised access to children's data

| Measure | Implementation |
|---|---|
| Multi-tenant isolation | Every database query scoped by group_id; no cross-tenant data access |
| JWT + session auth | Short-lived tokens (30 min access, 24h session) |
| Role-based access | Guardian, member, school_admin, club_admin roles enforced at API layer |
| Rate limiting | Redis sliding window (1000 req/min) + in-memory fallback |
| CORS enforcement | Production locked to bhapi.ai origins only |
| Security headers | CSP, HSTS, X-Frame-Options: DENY, X-Content-Type-Options: nosniff |

### R2: PII exposure in monitoring pipeline

| Measure | Implementation |
|---|---|
| PII detection engine | Regex + NER-based detection of names, emails, phones, addresses, SSNs |
| Auto-redaction | PII masked in stored content excerpts before persistence |
| Data minimisation | Raw prompts NOT stored; only risk-flagged excerpts retained |
| Content auto-deletion | 12-month TTL on all content excerpts |

### R3: Profiling/tracking concerns

| Measure | Implementation |
|---|---|
| No behavioural profiling | No user profiling, scoring, or automated decision-making |
| No advertising | Data never shared with advertisers or used for marketing |
| Purpose limitation | Processing limited to safety monitoring and spend management |
| Session cookie only | Single httponly secure cookie; no third-party tracking cookies |

### R4: Data breach impact

| Measure | Implementation |
|---|---|
| Encryption at rest | AES-256 via cloud provider (PostgreSQL) |
| Encryption in transit | TLS 1.2+ enforced (HSTS with 1-year max-age) |
| Credential encryption | Fernet (dev) / Cloud KMS (prod) for LLM API keys |
| Minimal data footprint | Only risk events stored, not full interaction history |
| Breach notification plan | 72-hour GDPR notification procedure documented |
| Penetration testing | Annual pen test by CREST-certified provider |

### R5: Guardian misuse

| Measure | Implementation |
|---|---|
| Age-appropriate transparency | Children aged 13+ can view their own monitoring dashboard |
| Consent requirements | COPPA (<13 US), GDPR (<16 EU), LGPD (<18 Brazil) consent required |
| Consent withdrawal | GDPR Art. 7(3) withdrawal endpoint available |
| Audit trail | All guardian actions logged in immutable audit log |

### R6: Inaccurate risk classification

| Measure | Implementation |
|---|---|
| Multi-tier classification | Keyword matching + optional Vertex AI/Gemini verification |
| Age-band sensitivity | Under-8 most conservative, 17+ most lenient |
| Guardian override | Guardians can acknowledge/dismiss false-positive risk events |
| Severity scaling | Critical/high/medium/low with different alert thresholds |

### R7: Sub-processor breach

| Measure | Implementation |
|---|---|
| Minimal data sharing | Stripe: billing only. SendGrid: email addresses only. |
| No child content shared | Sub-processors never receive AI interaction data |
| Vendor DPAs | Data Processing Agreements with all sub-processors |
| SCCs | Standard Contractual Clauses for international transfers |

### R8: Account takeover

| Measure | Implementation |
|---|---|
| bcrypt password hashing | Cost factor 12 |
| Password policy | Minimum 8 chars, must include upper/lower/digit |
| Rate-limited login | 10 attempts per minute |
| Password reset rate limit | 5 per hour per email |
| Session invalidation | Logout invalidates session token |
| MFA support | TOTP-based MFA available (optional) |

### R9: Payment data exposure

| Measure | Implementation |
|---|---|
| Stripe-hosted checkout | No card numbers touch Bhapi servers |
| Webhook HMAC validation | Stripe webhook signature verified |
| Test/live key separation | Stripe keys validated (sk_test_ vs sk_live_) |

## 6. Children's Data: Special Category Protections

### 6.1 Jurisdictional age thresholds

| Jurisdiction | Age Threshold | Regulation | Consent Required |
|---|---|---|---|
| United States | < 13 | COPPA | Verifiable parental consent |
| European Union | < 16 | GDPR Art. 8 | Parental consent |
| Brazil | < 18 | LGPD Art. 14 | Parental consent |
| Australia | < 16 | Privacy Act | Parental consent |
| United Kingdom | < 13 | UK GDPR + Age Appropriate Design Code | Parental consent |

### 6.2 Consent enforcement

- Capture events are blocked until guardian consent is recorded for the specific member
- Consent records include: type, timestamp, IP address, evidence reference
- Withdrawal is available at any time via API (GDPR Art. 7(3))
- Consent withdrawal immediately stops data processing for that member

### 6.3 Children's rights

- Right to be informed (age-appropriate privacy notices)
- Right of access (data export in machine-readable format)
- Right to erasure (account deletion cascades to all child data)
- Right to restrict processing (consent withdrawal per consent type)

## 7. International Transfers

| Data Flow | From | To | Safeguard |
|---|---|---|---|
| User data → Database | Worldwide | Cloud region (configurable) | Adequacy decision or SCCs |
| Email delivery | Worldwide | SendGrid (US) | SCCs + DPA |
| Payment processing | Worldwide | Stripe (US/EU) | SCCs + DPA |
| LLM spend sync | Worldwide | Provider APIs | No PII transferred; API keys only |

**Principle:** EU user data stored in EU region. AU user data stored in AU region. SCCs applied for all cross-border transfers where adequacy decisions are not in place.
